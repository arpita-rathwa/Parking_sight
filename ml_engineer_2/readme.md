# ParkSight ML — Illegal Parking Hotspot Detection

AI system for detecting illegal parking hotspots in Bengaluru and scoring their traffic impact, using spatial clustering (HDBSCAN), a composite impact score, and LightGBM-based validation. Includes three backend services (FastAPI scoring, Redis priority queue, Kafka event pipeline) for real-time dispatch.

---

## Project Structure

```
parksight-ml/
├── notebook/
│   └── parksight.ipynb          # main pipeline: clustering, scoring, ML validation
├── services/
│   ├── scoring_service.py       # FastAPI — serves impact-score predictions
│   ├── queue-service.py         # Redis — priority dispatch queue
│   └── kafka_service.py         # Kafka — producer/consumer wiring
├── model/
│   ├── model.txt                # trained LightGBM model
│   └── cluster_summary.json     # per-cluster features (used by queue-service)
├── data/
│   └── violations.csv           # raw violation records
├── outputs/
│   ├── hotspot_map.html         # interactive folium map of top hotspots
│   └── priority_queue.json      # ranked hotspot list
├── requirements.txt
└── README.md
```

## 1. Setup

### Install Python dependencies

```bash
cd parksight-ml
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

You'll know the venv is active when your terminal prompt shows `(venv)` at the start.

### Install Redis (macOS)

```bash
brew install redis
```

### Install Kafka (macOS)

```bash
brew install kafka
```

One-time storage format (only needed once, ever, per machine):

```bash
kafka-storage format -t $(kafka-storage random-uuid) -c /opt/homebrew/etc/kafka/server.properties
```

> **Intel Mac note:** if `/opt/homebrew` doesn't exist on your machine, use `/usr/local/etc/kafka/server.properties` instead in all Kafka commands below.

## 2. Running the project — full order

Open **4 terminal tabs**. Steps 1–3 are long-running services that must stay up; Step 4 creates a topic once.

### Terminal 1 — Redis

```bash
brew services start redis
redis-cli ping   # should print PONG
```

### Terminal 2 — Kafka broker

```bash
kafka-server-start /opt/homebrew/etc/kafka/server.properties
```

Leave this running — continuous log output is normal.

### Terminal 3 — FastAPI scoring service

```bash
cd parksight-ml
source venv/bin/activate
cd services
uvicorn scoring_service:app --host 0.0.0.0 --port 8000 --reload
```

Verify: open `http://localhost:8000/health` in a browser → should show `{"status":"ok"}`.

### Terminal 4 — One-time Kafka topic creation

```bash
kafka-topics --create --topic violation-events --bootstrap-server localhost:9092 --partitions 1 --replication-factor 1
kafka-topics --list --bootstrap-server localhost:9092   # should list "violation-events"
```

## 3. Run the notebook

With all three services above running, open and run `notebook/parksight.ipynb` **top to bottom**.

The notebook will, in order:
1. Load `data/violations.csv` and engineer time/severity/confidence features
2. Run HDBSCAN spatial clustering (~2,063 hotspot clusters)
3. Compute a composite `impact_score` per cluster and rank into `priority_queue`
4. Export `outputs/priority_queue.json` and `outputs/hotspot_map.html`
5. Train and validate a LightGBM model against the scoring formula
6. Save `model/model.txt` and `model/cluster_summary.json`
7. Test the live FastAPI `/score` endpoint with a real cluster
8. Push the top 20 hotspots into the Redis priority queue (`queue-service.py`)
9. Publish the top 10 hotspots to Kafka and consume them back

If a cell calling `localhost:8000`, Redis, or Kafka fails, double-check the corresponding terminal in Section 2 is still running.

## 5. Known notes

- `model/cluster_summary.json` is regenerated by the notebook (cell with `feature_table.to_json(...)`) — **run that cell before** running `queue-service.py`, or it'll use stale data.
- All notebook file paths are relative and assume `parksight.ipynb` stays inside `notebook/`. Don't move it without updating the `../data`, `../model`, `../outputs` paths.
- Kafka's one-time storage format (Section 1) only needs to run once per machine — running it again will error or reset existing topics.
